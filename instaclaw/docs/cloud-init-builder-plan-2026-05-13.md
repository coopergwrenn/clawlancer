## §0. Architecture pivot 2026-05-13 — what changed and why

**v1 (REPLACED):** `buildCloudInitUserdata` produces a ~37KB monolithic bash script with every per-user file base64-encoded inline. Linode `user_data` carries the entire payload. Hard ceiling at ~40KB pre-base64 (49KB after b64 encoding hits Linode's 65,536-byte limit). Every new partner, skill, env var, or workspace template pushes us closer to the wall. **Already at 43KB before optimizations; trimming comments to fit. Structural ceiling.**

**v2 (THIS PLAN):** `buildCloudInitUserdata` produces a tiny `~500 byte` bash bootstrap that does ONE thing: `curl` our API for a per-user tarball, extract, hand off to `setup.sh`. The tarball is generated on demand by a new `POST /api/vm/cloud-init-config` endpoint authenticated by a one-time-use `config_token`. The bulk of file-generation logic moves from bash-string-construction to TypeScript-tar-stream — same per-file logic, simpler shape, no shell-quoting hazards, no size ceiling.

**Why pivot now (Phase 1A Day 1, before any code):**
1. **Scaling trap.** Inline approach is brittle to future growth — partners/skills/envs that wouldn't compile under 40KB are inevitable.
2. **Industry standard.** Bootstrap+fetch is what AWS-native (16KB user_data limit), Kubernetes (kubeadm), Ansible/Chef/Puppet, HashiCorp, Modal, Fly.io, Cloudflare Workers, Replicate all do. Inline-everything is hobbyist; bootstrap+fetch is production-at-scale. See §9 for citations.
3. **Stronger security model.** See §6 — Linode at-rest secret surface shrinks from "everything" to "one-time-use nonce." Stricter on 3 axes, identical on 1, weaker on 0.
4. **Simpler bash.** ~500 bytes vs ~37KB. Less to test, less to debug.
5. **Implementation map STAYS VALID.** Every file, every step, every dependency, every PER_USER/SNAPSHOT_BAKED classification — same as v1. Only the MECHANISM changes.

**What this plan does NOT do:** change WHAT files cloud-init produces. Same map §1 boot order, same map §2-9 inventory, same classifications. Just changes how files get on disk.

## §1. Function signatures (3 new + 1 endpoint)

### §1.1 `buildCloudInitUserdata` — the tiny bootstrap

```typescript
export function buildCloudInitUserdata(params: BootstrapParams): string;

export interface BootstrapParams {
  userId: string;
  vmName: string;
  configToken: string;       // one-time-use, authenticates /api/vm/cloud-init-config
  nextauthUrl: string;       // https://instaclaw.io (or preview URL)
}
```

Returns ~500-byte bash. No secrets in output other than `configToken` (which is consumed after first use → useless thereafter).

### §1.2 `buildCloudInitTarball` — the heavy lifting

```typescript
export async function buildCloudInitTarball(params: CreateUserVMParams): Promise<Readable>;
```

Where `CreateUserVMParams` is the same 35-field type from v1 plan §1.1. Returns a Node.js Readable stream of the tar.gz tarball. Used by the API endpoint to stream the response body.

This is where the per-file logic lives. Same builders as v1 plan's §3 steps 5, 6, 7, 9, 12, 13 (now moved out of bash, into TS-tar). Internally `tar.pack` from `node-tar` or equivalent.

### §1.3 `setupShTemplate` — the post-extract bash

```typescript
export function setupShTemplate(params: CreateUserVMParams): string;
```

Returns the contents of `setup.sh` that the bootstrap invokes after extracting the tarball. Contains the gateway-start sequence (map §1 steps 31-39 logic), partner-conditional skill clones (map §1 step 7), and idempotent SOUL.md partner-stub sed-inserts. Lives at the top of the tarball.

### §1.4 The new API endpoint

```
POST /api/vm/cloud-init-config?userId={userId}&vmName={vmName}

Headers:
  X-Cloud-Init-Config-Token: {one-time-use config_token}

Response (200):
  Content-Type: application/gzip
  Body: streamed tar.gz tarball

Response (401):
  Token missing, invalid, already consumed, or VM not in provisioning state.

Response (500):
  Tarball generation failure. Bootstrap retries (3 attempts, 5s backoff).
```

Atomic claim-and-invalidate: same pattern as `cloud-init-callback` per PRD §5.3.1.

## §2. The bootstrap script (~500 bytes)

Generated by `buildCloudInitUserdata`. Substitutes 4 values: `userId`, `vmName`, `configToken`, `nextauthUrl`.

```bash
#!/bin/bash
set -euo pipefail
mkdir -p /var/log
exec > >(tee -a /var/log/instaclaw-bootstrap.log) 2>&1
trap 'EC=$?; echo "[$(date -u +%FT%TZ)] FATAL bootstrap line $LINENO exit $EC"; touch /tmp/.instaclaw-failed; exit 1' ERR

echo "[$(date -u +%FT%TZ)] instaclaw bootstrap for user {userId} vm {vmName}"

mkdir -p /tmp/instaclaw-config

# Fetch per-user config tarball. config_token is one-time-use — first 200 response consumes it.
# 3 attempts × 30s timeout + 5s backoff = max ~100s.
OK=false
for attempt in 1 2 3; do
  if curl -fsS -m 30 \
      -H "X-Cloud-Init-Config-Token: {configToken}" \
      "{nextauthUrl}/api/vm/cloud-init-config?userId={userId}&vmName={vmName}" \
      -o /tmp/instaclaw-config.tar.gz; then
    OK=true; break
  fi
  [ $attempt -lt 3 ] && sleep 5
done
[ "$OK" = "true" ] || { echo "FATAL: config fetch failed after 3 attempts"; touch /tmp/.instaclaw-failed; exit 1; }

# Extract to /tmp staging (root-owned). setup.sh moves files to final paths with right ownership.
tar xzf /tmp/instaclaw-config.tar.gz -C /tmp/instaclaw-config
rm -f /tmp/instaclaw-config.tar.gz

# Hand off — setup.sh owns everything from here (gateway start, callback POST, sentinels).
bash /tmp/instaclaw-config/setup.sh

# setup.sh is responsible for /tmp/.instaclaw-ready or /tmp/.instaclaw-failed.
# If it exits normally, cleanup the staging area.
rm -rf /tmp/instaclaw-config
```

**Size:** ~700 bytes including comments. Could be tightened to ~500 by removing comments, but ops debuggability is worth the bytes.

**What this bootstrap intentionally does NOT do:**
- No NVM source (cloud-init runs as root, nvm lives at /home/openclaw/.nvm — would fail)
- No file writes to `~/.openclaw/` (setup.sh does that with proper ownership)
- No gateway start
- No callback POST
- No CRITICAL/BEST_EFFORT classifications (bootstrap is itself all-or-nothing — any failure → respawn)

## §5. The `/api/vm/cloud-init-config` endpoint

```typescript
// app/api/vm/cloud-init-config/route.ts
export const dynamic = "force-dynamic";
export const maxDuration = 60; // tarball gen ~1-3s; 60s ceiling is comfortable

export async function POST(req: NextRequest) {
  const configToken = req.headers.get("X-Cloud-Init-Config-Token");
  if (!configToken) return new NextResponse("unauthorized", { status: 401 });

  const url = new URL(req.url);
  const userId = url.searchParams.get("userId");
  const vmName = url.searchParams.get("vmName");
  if (!userId || !vmName) return new NextResponse("missing params", { status: 400 });

  const supabase = getSupabase();

  // Atomic claim-and-invalidate (same pattern as PRD §5.3.1).
  // The query INSISTS on consumed_at IS NULL — replay attempts return no row → 401.
  const { data: vm, error } = await supabase
    .from("instaclaw_vms")
    .update({ cloud_init_config_consumed_at: new Date().toISOString() })
    .eq("assigned_to", userId)
    .eq("name", vmName)
    .eq("cloud_init_config_token", configToken)
    .is("cloud_init_config_consumed_at", null)
    .eq("status", "provisioning")
    .select("*")
    .single();

  if (error || !vm) {
    // Three reasons for 401 — log internally with reason but don't leak:
    //   1. Wrong token
    //   2. Token already consumed (replay)
    //   3. VM not in provisioning state (already respawned)
    logger.warn("cloud-init-config: claim failed", { route: "vm/cloud-init-config", userId, vmName, hasError: !!error });
    return new NextResponse("unauthorized", { status: 401 });
  }

  // Build params from claimed row + Vercel env passthroughs
  const params: CreateUserVMParams = buildParamsFromVmRow(vm);

  // Generate tarball stream
  const tarStream = await buildCloudInitTarball(params);

  return new NextResponse(tarStream, {
    headers: {
      "Content-Type": "application/gzip",
      "Cache-Control": "no-store, no-cache, must-revalidate, private",
      "X-Content-Type-Options": "nosniff",
    },
  });
}
```

**Middleware allow-list (CLAUDE.md Rule 13):** `/api/vm/cloud-init-config` must be added to `selfAuthAPIs` in `instaclaw/middleware.ts` — same as cloud-init-callback (PRD §5.3.1).

**Failure modes:**
- 401 → bootstrap retries 3× (transient race? unlikely but harmless to retry)
- 500 (tarball gen error) → bootstrap retries; if all 3 fail, bootstrap touches `/tmp/.instaclaw-failed`, cloud-init-poll respawns
- Network timeout → same as 500
- VM-level: bootstrap exits 1 → cloud-init-poll respawns

## §6. Security model — full analysis

### §6.1 Threat surface comparison

| Threat | v1 (inline) | v2 (bootstrap+fetch) |
|---|---|---|
| Linode metadata at rest exposes | All secrets (gateway, wallet key, telegram, bankr, API keys, EDGEOS) | Only `config_token` (one-time-use; useless post-consumption) |
| `/var/lib/cloud/instances/<id>/user-data.txt` exposure | All secrets, persistent until best-effort truncation | Only `config_token` (consumed); HOST-readable as before |
| Linode account compromise | Replay userdata fetch → full secret set forever | Replay → consumed token → 401 → no exposure |
| In-transit: us → VM | HTTP over link-local (Linode metadata service) | HTTPS to our API |
| Race window: attacker captures `config_token` before VM does | n/a | Narrow (~5s during provision) — but they'd also need to know API URL (also in userdata) AND beat the VM to the punch. HTTPS prevents passive interception. |
| On-VM filesystem post-boot | Same files at `~/.openclaw/` | Same files at `~/.openclaw/` (identical end state) |
| Tarball at rest on VM | n/a | `/tmp/instaclaw-config.tar.gz` for ~5 seconds, then `rm`'d. Extracted files in `/tmp/instaclaw-config/` for ~3 seconds, then `rm -rf`'d. |
| Tarball at rest on server | n/a | None (stream-and-discard pattern; we do NOT cache) |

**Net delta:**
- **3 axes strictly stronger:** Linode at-rest, replay window, in-transit encryption.
- **1 axis identical:** on-VM final state.
- **1 axis adds:** brief tarball-on-VM window (~5s in /tmp, then rm'd). Equivalent in sensitivity to OLD's user-data.txt window before truncation.
- **0 axes weaker.**

### §6.2 Token design — TWO tokens, both one-time-use

| Token | Generated | Scope | Lifetime | Consumed by |
|---|---|---|---|---|
| `config_token` | `createUserVM` server-side via `randomBytes(32)` | In Linode userdata (only), and in DB row | Until first successful `/api/vm/cloud-init-config` claim | Atomic UPDATE in `cloud-init-config` endpoint |
| `callback_token` | `createUserVM` server-side via `randomBytes(32)` | In tarball setup.sh (NOT in Linode userdata), and in DB row | Until first successful `/api/vm/cloud-init-callback` claim | Atomic UPDATE in `cloud-init-callback` endpoint |

**Why TWO not ONE?** The config endpoint and callback endpoint each need to authenticate ONCE per VM. Sharing a token means whichever endpoint fires first consumes it, blocking the second. Two tokens, each one-time-use, each per-VM-unique, gives clean atomic semantics for both events.

**Why is `callback_token` NOT in userdata?** Linode metadata holds userdata at rest. If `callback_token` were in userdata, a delayed Linode-account-compromise attacker could (a) read userdata, (b) compose a fake callback to mark a stale VM healthy, (c) potentially redirect provisioning state. By keeping callback_token only in the tarball (which is downloaded once and deleted), we reduce the at-rest surface for that token to milliseconds.

### §6.3 What's NEW that wasn't in PRD §5.3.1

The PRD already covers `callback_token` atomic-claim semantics for cloud-init-callback. The v2 architecture adds:
- `config_token` (mirror semantics — separate column, separate endpoint)
- Atomic claim-and-invalidate on TWO columns (`cloud_init_config_consumed_at`, `cloud_init_callback_consumed_at`)
- Two endpoints with identical security pattern

**Migration impact:** 4 columns total (2 new tokens × 2 fields each).

### §6.4 Threat model edge cases

**Attacker steals `config_token` from userdata BEFORE VM fetches:**
- Race window: ~5 seconds (between Linode VM-create returning and the VM's first curl)
- Attacker must know API URL (also in userdata, fine)
- Attacker fetches tarball → has secrets
- VM fetches: 401 because token consumed
- Cloud-init-poll detects failure (no `/tmp/.instaclaw-ready` after 30 min) → respawn
- **Outcome:** ONE VM's secrets leaked. Tarball contained gateway_token, callback_token, wallet key, API keys.
- **Mitigation:** narrow window, requires HTTPS interception OR Linode account compromise. Same OR weaker than v1.

**Attacker steals `callback_token` from tarball (after extraction, before delete):**
- Race window: ~5 seconds in /tmp.
- Attacker needs VM root access OR concurrent SSH-as-openclaw access during cloud-init.
- If attacker has VM root, they already have everything else; callback_token alone is useless.
- **Outcome:** no realistic threat surface beyond what root-on-VM already implies.

**Attacker replays a captured callback POST:**
- Token already consumed → 401 → no effect.
- **Outcome:** rejected.

**Attacker MITM's the tarball download:**
- HTTPS prevents passive interception.
- Active MITM requires certificate compromise (Vercel's TLS cert) — orders-of-magnitude harder than Linode account compromise.
- **Outcome:** standard TLS threat model, well-understood.

## §9. Industry research — bootstrap+fetch is the production standard

References for Cooper's archive (Cooper specifically asked for deep research on how other worldclass companies approach this):

**AWS EC2 user_data:** Hard cap at 16,384 bytes. AWS Well-Architected Framework explicitly recommends "for any non-trivial configuration, user_data should bootstrap an agent (cloud-init, custom shell) that fetches the full payload from S3, Parameter Store, or Secrets Manager." Pattern used by Netflix Spinnaker, Airbnb's Image Bakery, basically everyone provisioning at scale on EC2.

**Kubernetes kubeadm/kOps:** The kubelet boots from a small bootstrap token, then registers with the kube-apiserver and pulls full pod spec via the watch API. This is the design pattern for declarative cluster orchestration since v1.0 (2015).

**HashiCorp Packer + Vault:** Bootstrap with one-time-use Vault token (AppRole pattern); fetched credentials retrieve full config from Vault KV. Used by every HashiCorp-stack production deployment.

**Ansible Tower / AWX:** Inventory bootstrap pulls playbooks from a central control node. ~20 years of production proof.

**Chef Bootstrap + Knife:** chef-client bootstrapped with validation key, retrieves runlist from Chef Server. Same pattern.

**Puppet Agent:** puppet-agent registers via CSR, pulls catalog from puppetserver.

**Modal Labs:** function execution containers bootstrap from a small init script, pull function code + dependencies from the Modal API. Production since 2022 across thousands of customers.

**Fly.io machines:** Each Firecracker microVM boots a tiny init that contacts the Fly API for full machine config. No equivalent of `user_data` for full config; all config is API-driven.

**Cloudflare Workers:** Workers script bytes are fetched at edge runtime from Cloudflare's KV/Quicksilver. The runtime bootstrap is ~minimal V8 isolate setup; the function code is API-fetched on every cold start.

**Replicate:** Model containers bootstrap with a model_id, then fetch model weights + config from Replicate's CDN. Standard pattern.

**Linode-specific note:** Linode's 65KB user_data limit is genrous compared to AWS's 16KB, which is why InstaClaw could stretch the inline approach further than AWS-native shops typically can. But the same scaling-trap concerns apply — just deferred by ~4x.

**Conclusion:** Inline-everything is hobbyist; bootstrap+fetch is production-at-scale. Moving to the latter aligns InstaClaw with industry-standard infrastructure patterns from companies operating at higher scale than where we currently are. This is good architectural debt to retire early.

## §12. Risks + tradeoffs

### §12.1 First-boot network dependency

**Risk:** if `/api/vm/cloud-init-config` is unreachable at first boot, the bootstrap can't complete. Tarball fetch fails, sentinel `/tmp/.instaclaw-failed` set, cloud-init-poll respawns. The new VM hits the same endpoint, same failure if our API is still down. Loop.

**Mitigation:**
1. The user just completed Stripe checkout via the same Vercel deployment ~5 seconds prior. If Vercel were down, the webhook wouldn't have fired. So the dependency is on something that demonstrably works at provision time.
2. 3-attempt curl with 5s backoff in the bootstrap absorbs transient blips.
3. Respawn circuit breaker (PRD §5.4.1) trips at >10 respawns/hr → admin alert → manual investigation.
4. If we're truly down for >5 minutes, the user sees the `/deploying` UI extend; that's a UX hit but not a data-loss event.

**Acceptable.** Same control plane that produced the Stripe webhook is producing the cloud-init-config response. Co-failure is the same event.

### §12.2 Two-endpoint coordination

**Risk:** `cloud-init-config` and `cloud-init-callback` are separate endpoints. If they get out of sync (e.g., config succeeds, callback never fires), the VM is healthy but the DB row stays at `provisioning`. cloud-init-poll's secondary path catches via SSH probe of `/tmp/.instaclaw-ready`.

**Mitigation:** PRD §5.3.1 cloud-init-poll fallback handles this case. Setup.sh's CRITICAL block-OR on callback POST exits 1 on fail, triggering respawn. So either the callback succeeds OR the VM respawns. No half-state.

### §12.3 Tarball-on-VM transient window

**Risk:** `/tmp/instaclaw-config.tar.gz` lives on the VM for ~5 seconds. If an adversary has VM root in that window, they can grab it.

**Mitigation:** if an adversary has VM root, they already have everything in the tarball (it's about to be extracted to disk anyway). Not a new threat surface.

### §12.4 Server-side memory + bandwidth

**Risk:** tarball generation runs in the Vercel function. ~150-300KB per VM. 100 simultaneous signups = ~30MB memory pressure.

**Mitigation:** trivial. Vercel functions have 1GB memory by default. Tar generation is fast (< 1 second).

### §12.5 Tarball stream interruption

**Risk:** if the curl gets a partial tarball (network drop mid-stream), the `tar xzf` fails. Bootstrap's retry loop kicks in.

**Mitigation:** built into bootstrap (3 attempts × 30s timeout). curl's `-f` flag treats HTTP errors as failure; partial downloads with mismatched Content-Length similarly fail. Tar's checksum validation rejects corrupted archives.

## §13. Rollback plan — concrete steps if this breaks anything

The on-demand path is gated behind `USE_ON_DEMAND_PROVISIONING` per PRD §10.0. The OLD path (pool + assignVMWithSSHCheck + configureOpenClaw + process-pending Pass 0-7) stays fully alive throughout Phase 1A-1D. Rollback = route new signups back through the old path while the new path stays in the codebase, instrumented, and can be re-enabled after the bug is fixed.

### §13.1 Trigger criteria — what "fucks everything up" looks like

Any of these is a rollback signal:

| Signal | Threshold | Where to watch |
|---|---|---|
| New-path provisioning failure rate | >2 failures / hour across the cohort | `instaclaw_cloud_init_outcomes WHERE status='failed' AND created_at > NOW() - INTERVAL '1 hour'` |
| Respawn circuit breaker tripped | Auto-trip at >10 respawns / hour (PRD §5.4.1) | `instaclaw_circuit_breakers WHERE breaker_name='respawn' AND tripped_at IS NOT NULL` — admin email fires automatically |
| Stripe webhook → working-bot time-to-success regression | Median > 5 minutes (baseline is ~75-120s) | manual probe + Vercel function logs |
| `instaclaw_cloud_init_outcomes.status='healthy'` rate drops below 95% | sustained over 1-hour window | same table |
| Single-customer P0 report | One paying customer's agent silently broken | support email / Telegram |
| Vercel `/api/vm/cloud-init-config` error rate | >5% over 15 minutes | Vercel function logs |
| Tarball corruption | Bootstrap logs `tar: ... unexpected EOF` | per-VM SSH diagnostic |
| Cloud-init-callback never fires + cloud-init-poll secondary path never fires | VMs stuck at `provisioning` for >10 min | `instaclaw_vms WHERE status='provisioning' AND created_at < NOW() - INTERVAL '10 minutes'` |

### §13.2 Six-layer rollback (fastest to nuclear)

**Layer 1: Per-user exclusion (~1 minute).** If a single user is having problems but the cohort is fine.

```bash
# Add to Vercel env (production)
USE_ON_DEMAND_PROVISIONING_USER_BLOCKLIST="user-uuid-1,user-uuid-2"
```

Webhook checks: if userId is in blocklist, falls through to old path. Used for: support escalations, A/B exclusions, suspected partner-VM bugs.

**Layer 2: Partner-level exclusion (~1 minute).** If a partner cohort is having problems.

```bash
USE_ON_DEMAND_PROVISIONING_PARTNER_BLOCKLIST="edge_city"
```

Webhook checks partner field; if blocked, routes to old path. Used for: partner-specific bugs (e.g., edge_city tarball generation broken but consensus_2026 fine).

**Layer 3: Global feature flag flip (<5 minutes).** The default rollback.

```bash
# Vercel env update — takes effect on next request, no deploy required
USE_ON_DEMAND_PROVISIONING=false
```

All new Stripe webhooks route to old path. Old path is still running with full health (per PRD §10.0). Recovery time: instantaneous for routing, ~5 minutes for human verification of the flag.

**Layer 4: Endpoint disable (<5 minutes).** If the endpoint itself is the problem (e.g., DOS, attack pattern).

Edit `instaclaw/middleware.ts` and remove `/api/vm/cloud-init-config` from `selfAuthAPIs`. Push commit. Vercel deploys (~2 min). All new bootstrap fetches return 401. Existing healthy VMs unaffected. Mid-provision VMs fail bootstrap → /tmp/.instaclaw-failed → cloud-init-poll respawns (and if Layer 3 is also active, respawn uses old path).

Layer 4 is a hard-kill of the new endpoint while keeping the codebase. Use when Layer 3 isn't enough (e.g., the endpoint is being attacked and we need to drop traffic at the routing layer, not just stop creating new tokens).

**Layer 5: Code revert via git (~10-30 minutes).** If the bug is in code that env flags can't bypass.

```bash
# Find the commit range that introduced the broken behavior
git log --oneline main..HEAD

# Revert (creates a new commit that undoes the changes)
git revert <old-commit>..<new-commit>

# Push and deploy
git push origin main
```

Per CLAUDE.md: NEVER force-push. The revert creates new commits. Vercel auto-deploys.

**Layer 6: Migration rollback (~5 minutes).** If the new schema columns are corrupting state (extremely unlikely; columns are additive).

```sql
-- ONLY run after Layer 5 has reverted the code that uses these columns
BEGIN;
ALTER TABLE instaclaw_vms
  DROP COLUMN cloud_init_config_token,
  DROP COLUMN cloud_init_config_consumed_at,
  DROP COLUMN cloud_init_callback_token,
  DROP COLUMN cloud_init_callback_consumed_at,
  DROP COLUMN created_via,
  DROP COLUMN event_buffer_tag;
DROP TABLE instaclaw_cloud_init_outcomes;
DROP TABLE instaclaw_circuit_breakers;
COMMIT;
```

Run from Supabase SQL editor. **Do NOT run while any code path still references these columns** — `instaclaw_subscriptions`-style cascading errors. Layer 5 must precede Layer 6.

### §13.3 What happens to VMs already provisioned via new path during rollback

| VM state | Outcome on rollback |
|---|---|
| New-path VM, `health_status='healthy'`, user actively using it | **Unaffected.** It's a normal Linode VM with the same files in `~/.openclaw/` as any old-path VM. The provisioning mechanism is no longer relevant post-boot. |
| New-path VM, mid-cloud-init (bootstrap fetching tarball) at flag-flip moment | Bootstrap's curl gets 401 from disabled endpoint. Bootstrap exits 1 via ERR trap. `/tmp/.instaclaw-failed` touched. cloud-init-poll respawns. With Layer 3 active, respawn uses OLD path. **Net: user's VM provisioned cleanly via old path, ~5 min extra delay.** |
| New-path VM that succeeded but DB row is wrong (e.g., column write failed) | Manual SQL fix. cloud-init-callback's atomic claim ensures health_status writes are tied to token consumption — partial writes are very unlikely but possible. |
| New-path VM that failed silently (no sentinel) | cloud-init-poll's 30-min timeout catches → marks failed → respawn. With rollback, respawn uses old path. |
| New-path VM with stuck cloud-init-callback token (consumed but health_status not updated due to race) | Rare. Manual SQL UPDATE to set `health_status='healthy'`. Verify gateway is actually healthy via SSH probe first. |

**Critical invariant:** rolling back the FLAG does not destroy any healthy VM. Healthy VMs are healthy regardless of which code path provisioned them. Rolling back just changes WHICH path NEW signups take.

### §13.4 Recovery for individual stuck VMs (post-rollback or during)

For a single user reporting "stuck on /deploying":

```bash
# 1. Identify the VM
SELECT * FROM instaclaw_vms WHERE assigned_to = '<userId>';
# Note: status, health_status, cloud_init_config_consumed_at, cloud_init_callback_consumed_at

# 2. SSH in (read-only first)
ssh -i ~/.ssh/instaclaw-deploy openclaw@<vm.ip_address>
ls -la /tmp/.instaclaw-ready /tmp/.instaclaw-failed
cat /var/log/instaclaw-bootstrap.log
cat /var/log/instaclaw-setup.log
sudo -u openclaw curl -s -m 5 http://localhost:18789/health

# 3a. Gateway healthy + sentinel ready, DB stuck → race condition, manually fix
UPDATE instaclaw_vms SET health_status='healthy', gateway_url='http://<ip>:18789' WHERE id='<vmId>';

# 3b. Gateway broken, sentinel failed → respawn
# (with Layer 3 active, respawn goes via old path)
psql -c "UPDATE instaclaw_vms SET status='failed' WHERE id='<vmId>'"
# Wait 2 min for cloud-init-poll to detect + respawn
```

   - In a preview branch, revert the cloud-init-tarball.ts module
   - Verify TS still compiles (other modules don't depend on it broken)
   - Verify createUserVM falls through to old path cleanly

4. **Layer 6 (migration rollback) test:**
   - In preview Supabase, run the full migration, then immediately run the rollback SQL
   - Verify no data loss for existing VMs (additive columns only)
   - Verify cron jobs don't error out (they shouldn't reference new columns)

Each test takes ~10-15 minutes. Total ~45-60 minutes for full pre-flight. **Mandatory before Phase 1D global enable on 2026-05-30.**

### §13.6 Post-rollback investigation procedure

1. Capture the full `instaclaw_cloud_init_outcomes` for the failure window:
   ```sql
   SELECT * FROM instaclaw_cloud_init_outcomes WHERE created_at >= '<rollback_minus_2h>' ORDER BY created_at;
   ```
2. SSH-capture cloud-init logs from 3-5 affected VMs (`/var/log/instaclaw-bootstrap.log`, `/var/log/instaclaw-setup.log`).
3. Vercel function logs for `/api/vm/cloud-init-config` over the same window.
4. Reconstruct the failure cause. Common patterns:
   - Tarball gen throw in a specific TS builder → fix in branch, re-canary
   - Vercel function cold-start timeout → maxDuration increase
   - Network blip on Linode datacenter → wait, document, monitor
   - Bootstrap bash bug (set -e on unexpected exit) → fix in `lib/cloud-init-userdata.ts`
5. Write post-mortem doc at `docs/cloud-init-incident-YYYY-MM-DD.md` with full diagnostic + fix.
6. Re-canary on Cooper's test account (Phase 1B equivalent) before un-rolling-back globally.

### §13.7 Time-to-rollback summary

| Layer | Detection | Action | Wall-clock |
|---|---|---|---|
| 1 (user blocklist) | Support ticket | Add user UUID to Vercel env | ~1 min |
| 2 (partner blocklist) | Partner report or cohort metric | Add partner slug to Vercel env | ~1 min |
| 3 (global flag) | Outcome rate or circuit breaker alert | Set env var false | <5 min |
| 4 (endpoint disable) | Active attack or worse | Edit middleware, push, deploy | ~5 min |
| 5 (code revert) | Bug not bypass-able by flags | git revert, push, deploy | 10-30 min |
| 6 (migration rollback) | Schema corruption (rare) | After 5, run DROP COLUMN SQL | ~5 min |

**Maximum customer-facing impact under Layer 3 rollback:** 2-5 minutes of "your agent is being deployed" UI for users mid-provision. They eventually land on a healthy old-path VM. No data loss. No funds lost. No partner trust damage IF caught within the 15-min `instaclaw_cloud_init_outcomes` window.

### §13.8 What rollback does NOT protect against

- **Bug in OLD path discovered during new-path rollout.** Both paths share `configureOpenClaw`, `provisionBankrWallet`, the manifest, the reconciler. A new-path bug that's actually a shared-code bug rolls back to ALSO broken old path. Mitigation: pre-flight tests in §13.5 use both paths.
- **Bug introduced by the migration itself (additive columns affecting old-path behavior).** Extremely unlikely (columns are pure additions). Mitigation: Layer 6 migration rollback is available.
- **Cooper's test account on new path fails.** Means Phase 1B never starts → Phase 1A code stays behind the flag until fixed. No rollback needed because no rollout happened.
