## Context

Per the call summary in `docs/prd/edge-esmeralda-master-prd-2026-05-19.md`, Edge Esmeralda routes through a two-layer architecture:

- **Layer 1: `instaclaw.io/edge`** (Seref's redesign, currently live) — cinematic marketing page with a single CTA pill ("CLAIM YOUR AGENT"). No claim form, no consent, no email widget. Its sole job is to make the visitor want the next page.
- **Layer 2: `instaclaw.io/edge/claim`** (this proposal) — the tactical setup page. Direct, action-focused, accessible from the Layer 1 CTA and from the persistent `CLAIM AGENT` pill in the nav.

This proposal describes only Layer 2. Layer 1's redesign is owned by Seref and shipped separately.

## Research summary

### What `/edge-v1-backup` already had (still in repo, deployed at `/edge-v1-backup`)

- A ~767-line landing page covering hero + stat strip + overnight loop + features + privacy posture + research note + FAQ + sponsors + final CTA + footer.
- The hero hosted `EdgeCityClient`, a single client component that already handles all three user states (`logged_out`, `in_progress`, `live`):
  - `logged_out`: claim-CTA (POSTs `/api/partner/tag` then routes to `/signup`) + "or" divider + email notify-me form (POSTs `/api/notify` with `source: "edge_city"`).
  - `in_progress`: "Complete setup →" pill routing to `state.resumePath`.
  - `live`: sage card with pulsing dot, `@botUsername`, "Open in Telegram →" deep link.
- Privacy section enumerated the 4 principles now living on `/edge/consent` (verbatim source).
- The visual signature: tracked-out caps for eyebrows, huge uppercase serifish display font for headings (e.g. `AGENT VILLAGE.`), dotted-grid background, dark-olive primary CTAs on cream background, sage accents.

### What the existing BYOK flow does (`app/(onboarding)/plan/page.tsx`)

- Plan page has a sliding "All-Inclusive ↔ BYOK" toggle on the tier-pricing area (lines 433-485).
- BYOK = "Bring Your Own Anthropic API Key — lower monthly cost, you pay Anthropic directly." Tier prices halve to a third when BYOK is on (e.g. $99 Pro all-inclusive → $39 Pro BYOK).
- BYOK selection persists into `pending_users.api_mode` and ultimately drives `auth-profiles.json` shape on the VM (proxy token for all-inclusive, direct Anthropic SK for BYOK).

### What Cooper meant by "BYO branch" on `/edge/claim`

After re-reading the call summary and CLAUDE.md Rule 9, this is NOT the BYOK pricing toggle — Edge attendees don't pay either way (inference is sponsor-funded for the village). "BYO" here means: "I already have an InstaClaw account and an agent. Sign me in and tag my existing agent as `edge_city` instead of provisioning a new one."

The mechanism already exists — `POST /api/partner/tag` updates `instaclaw_users.partner` on the logged-in user's record and syncs `instaclaw_vms.partner` on their assigned VM (per Rule 9). The `EdgeCityClient.handleClaim()` function already calls this endpoint for both logged-in and logged-out users.

**The gap:** logged-out users with an existing account currently see only "Claim your agent →" which routes to `/signup`. They have to know to click "Sign in" on that page. We can do better by surfacing the existing-account path explicitly on `/edge/claim`.

## Design goals

1. **Action-focused.** This page exists to convert. One primary CTA, two secondary affordances. No marketing sections.
2. **Brand-seam continuity.** Inherits `/edge/layout.tsx`'s palette (cream / forest / sage / olive). Visual vocabulary matches v1-backup hero so a visitor scrolling from `/edge` to `/edge/claim` feels like they're walking through one document, not switching products.
3. **Mobile-first.** ≥48px tap targets. Thumb-reachable primary CTA. 16-18px body text. No horizontal scroll at 390px.
4. **Trust at the surface.** Sponsor-funded badge + consent-brief link sit immediately under the form so researchers and skeptics see the integrity disclosure inline with the conversion ask.
5. **Audience-aware.** Cypherpunks + researchers don't want enthusiasm copy. Direct, factual, slightly technical.

### 3. Copy register (every word weighed)

- **Headline**: `CLAIM YOUR AGENT.` — uppercase display serif (matching v1-backup pattern). Three words. "Claim" carries scarcity + ownership (Cooper's note). Period as punctuation, not interrobang.
- **Body** (≤45 words):

  > Your personal agent for the 28-day village. One Telegram message every morning. Free for verified ticket holders.

- **Primary CTA label**: `CLAIM YOUR AGENT →` for logged_out, `COMPLETE SETUP →` for in_progress, sage live-state card with `@botUsername` for live (all three reuse EdgeCityClient verbatim — no copy changes there).
- **Consent line**:

  > By claiming you agree to the EE26 research program. [Read the consent brief.](/edge/consent)

- **BYO branch**:

  > Already have an InstaClaw agent? [Sign in to claim it for Edge →](/login?callbackUrl=/api/partner/tag-redirect)

  (See "BYO branch wiring" below — there's a small route to add.)

- **Trust band**:

  > Free for verified ticket holders · Inference sponsor-funded · Anonymized research

  Single line, 11px tracked caps, border-top hairline. Same line as v1-backup but compressed.

### 4. EdgeCityClient — reuse, do not rewrite

`app/edge-v1-backup/edge-city-client.tsx` already implements every behavior we need:

- claim CTA → POSTs `/api/partner/tag` → routes to redirect_to (`/dashboard` if existing user; `/signup` if new).
- notify-me form → POSTs `/api/notify` with `source: "edge_city"`.
- live-state card.
- in-progress pill.

The current `app/edge/edge-city-client.tsx` (used by Layer 1's Claim section) is an updated copy of the v1-backup client — already in production, already wired to the same APIs.

For `/edge/claim`: **import and reuse** `EdgeCityClient` from `app/edge/edge-city-client.tsx`. No code duplication.

### 5. BYO branch wiring (new — small surface)

The current `/api/partner/tag` returns `redirect_to` based on session state, but only when called directly. The "sign in to claim existing" flow needs an intermediate step:

1. User clicks `Sign in to claim it for Edge →` from `/edge/claim`.
2. We navigate to `/login?callbackUrl=/api/partner/tag-redirect`.
3. `/api/partner/tag-redirect` (new route, ~20 lines): server-side handler that calls the same partner-tag logic, then 302s to `/dashboard`.
   - Setting `partner=edge_city` from a logged-in-during-OAuth-callback request closes the Rule 9 gap (the cookie-only flow doesn't tag existing users).
4. Result: existing users land back on `/dashboard` with their account tagged + VM synced.

This new route is the only net-new server-side surface. Everything else is composition + copy.

### 6. Not on `/edge/claim`

Explicit list of what we do NOT add — keeping the page focused:

- No overnight-loop section (lives on `/edge` Layer 1).
- No features grid.
- No FAQ accordion.
- No research narrative card.
- No sponsor strip.
- No stat band (`~500 / 28 / 0 / 1`).
- No second CTA at bottom.
- No nav with multiple links (back-to-`/edge` is enough).

Anyone wanting context lands here and stays focused on the one action. Anyone wanting context bails back to `/edge` and reads the marketing page.

### FUP-1: Duplicate-agent prevention on /edge/claim primary CTA — **MONEY-CRITICAL**

When a logged-in InstaClaw user clicks **CLAIM YOUR AGENT** on /edge/claim,
they currently flow through the new-signup path. If they already have a
VM, we just provisioned a second one (~$29/mo and ~3 min reconciler
overhead per ghost agent). Cooper's framing: "every duplicate agent we
prevent is $29/mo in VM costs we don't eat."

State-aware routing on the primary CTA (smart click handler):

- `userState.kind === "logged_out"` → current behavior. Set cookie,
  send to /signup → /connect.
- `userState.kind === "live"` (logged in + has working VM) → route to
  `/api/partner/tag-redirect` (the GET handler already shipped). Tags
  the existing VM as `edge_city`, redirects to /dashboard. Surface a
  brief interstitial / toast: *"Your existing agent has been upgraded
  for Edge Esmeralda."*
- `userState.kind === "in_progress"` (logged in, no usable VM yet) →
  current behavior. They haven't completed signup — let them finish.

The secondary "Already have an InstaClaw agent? Sign in to claim it for
Edge →" link still serves users who are logged OUT but know they have
an account — that path is untouched.

**Implementation sketch:** EdgeCityClient already receives `userState`
prop. Add a branch in `handleClaim`: if `state.kind === "live"`, call
`router.push("/api/partner/tag-redirect")` BEFORE the POST to
`/api/partner/tag`. Otherwise same code path.

**ULTRATHINK the detection:** `getEdgeUserState()` returns `live` only
when the VM is fully usable (gateway_url + telegram_bot_username +
health_status ∈ healthy/hibernating/suspended). For `frozen` it returns
`in_progress.resumePath=/dashboard` — that user has an archived VM and
clicking CLAIM should route to /dashboard for thaw, NOT create a new
agent. Need to ALSO route the in_progress + resumePath=/dashboard case
through tag-redirect to avoid the "user has frozen VM, claims new
agent" duplicate-spend variant.
