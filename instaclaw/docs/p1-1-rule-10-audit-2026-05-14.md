### Footnotes — semantic-misclassification concerns (low-severity Tier 3 followups)

¹ **"skip" cases pushed to `alreadyCorrect`** instead of `result.warnings`. Examples: stepExecStartAlignment lines 1262/1271/1302, stepCaddyUIBlock line 3931. These detect "couldn't safely proceed" (e.g., no unit file, non-canonical ExecStart, no Caddyfile). The work was NOT already done; it WAS NEVER REQUIRED. Pushing to `alreadyCorrect` is semantically misleading in audit logs. **Does NOT cause lying-DB** because (a) every condition that implies real broken state is caught by a sibling step that pushes errors, (b) the cv bump gate uses `result.errors.length > 0`, not `alreadyCorrect`. Recommend: rename to `result.warnings.push(...)` for clarity. Tier 3 followup; not P1-1-blocking.

² **stepRemovePlaceholder line 2444**: python probe `... || echo "SKIP"`. If the python fails (file unreadable), output is "SKIP" not "NO", and the check `has === "YES"` falls through to the alreadyCorrect path. This means if openclaw.json is structurally broken, this step silently no-ops. However, `stepAuthProfiles` (line 3359) reads the same file via a different path and pushes errors on parse failure — so the broken state IS caught downstream. Acceptable.

³ **stepSystemPackages line 3200**: `build-essential` is a meta-package with no binary; `which build-essential` always returns MISSING, so the step retries `apt-get install build-essential` every cycle. Already documented as working-as-intended in CLAUDE.md v88 changelog. Wasteful but idempotent. Tier 3 followup: change check from `which` to `dpkg -l | grep build-essential` for meta-packages.

⁴ **stepClearProviderCooldown line 3335**: `CLEAN` is true-no-op; `SKIP` is "python failed" (file unreadable). Both push alreadyCorrect. `stepAuthProfiles` (line 3359) catches the unreadable-file case downstream. Acceptable.

⁵ **stepMigrateSoulV2 line 5526**: "no SOUL.md found, nothing to migrate". This should never fire in practice because `stepWorkspaceIntegrity` (line 849) creates SOUL.md from a minimal template if missing. If it ever does fire, would suggest stepWorkspaceIntegrity failed earlier. Better to push to `result.warnings` so it's visible. Tier 3 followup.
